⚙️Neural-ART™ Accelerator — STM32N657

Both the Nucleo-N657X0-Q and the STM32N6570-DK mount the same STM32N657 SoC and therefore share an identical NPU silicon — any performance difference observed between the two boards is purely due to compiler version and toolchain configuration.

Streaming Dataflow Architecture

The Neural-ART NPU uses a streaming processing engine: rather than loading full activation maps into SRAM, data flows continuously through the MAC array from input to output, dramatically reducing memory bandwidth requirements.

Hardware Epoch Concept

When a layer's activation tensors exceed the ~1.5 MB NPU SRAM buffer, the compiler partitions execution into multiple hardware epochs — tiling the computation so each epoch fits within the physical memory limit. More epochs → higher latency.

🔢Hardware Epoch Partitioning per Layer & Platform

What Drives Epoch Count?

The compiler partitions a layer into N epochs when the activation buffer required exceeds the internal ~1.5 MB SRAM window. Each additional epoch introduces a reload-and-store cycle, increasing latency.

1️⃣

Most standard convolutions (16f 3×3, 32f 3×3, 64f 3×3, 5×5) fit in a single epoch on all platforms — excellent NPU utilization.

2️⃣

Depthwise 3×3 is split into 2–3 epochs depending on the platform: the compiler over-allocates RAM due to weight-repacking for NPU alignment, forcing partitioning unnecessarily.

2️⃣

Pointwise 1×1 requires 2 epochs on most platforms despite low MACC count — its large output activation map (64×64×32) is the bottleneck.

4️⃣

Conv2D 32f 7×7 requires 4–5 epochs on v2.2 platforms due to 1,292–1,548 KiB activations; v4.0 times out entirely.

Estimation Accuracy Analysis

Δ% = (Actual − Estimated) / Estimated × 100
Positive = compiler under-estimates (risky — may cause OOM).
Near-zero = accurate prediction.

✅

Standard convolutions (3×3, 5×5, 64f 3×3): Δ% is ≈ 0% across all tools — the compiler predicts RAM correctly for typical compute-bound layers.

ℹ️

7×7 layer: estimation is accurate (near 0%) — the model is too large by design, not due to compiler error.

🚀Throughput — MACC/ms (NPU Utilisation Proxy)

Why Throughput Matters

Raw inference time is model-size-dependent. Throughput (MACC/ms) normalises by complexity and reveals true NPU utilisation — how many operations per ms the hardware is actually executing.

🏆

CubeMX v2.2 delivers the highest throughput for compute-bound layers like 5×5 and 7×7, suggesting superior operator scheduling and NPU utilisation by the v2.2 compiler backend.

📉

Pointwise 1×1 and Depthwise 3×3 show dramatically lower throughput on all platforms — these layers are memory-bound, not compute-bound, and the streaming engine cannot fully pipeline due to large output tensors.

📊

Standard Conv (3×3, 5×5, 64f 3×3) achieve the best MACC/ms — close to the theoretical peak — confirming that regular convolutions are the NPU's "sweet spot" operator.

🔍NPU Behaviour — Key Findings

① Compiler Generation Matters More Than Board

The same STM32N657 silicon shows very different performance depending on AI Core version. v2.2 (CubeMX) outperforms v4.0 (AI Studio) on the same Nucleo board for most layers — suggesting v4.0 epoch scheduling is not yet fully optimized for all operator types.

② 1.5 MB SRAM is the Hard Limit

The internal NPU SRAM buffer (~1.5 MB) is the primary constraint. Layers with activation tensors exceeding this require multi-epoch partitioning (latency penalty) or fail entirely (v4.0 Epoch Controller cannot handle 7×7 even with tiling).

③ Standard Conv2D is the NPU Sweet Spot

Regular convolutions (3×3, 5×5, 64f 3×3) show near-linear MACC→latency scaling, excellent throughput, accurate RAM predictions, and single-epoch execution. The Neural-ART streaming engine is purpose-built for these patterns.

④ Memory-Bound Layers are Problematic

Pointwise 1×1 (large output activations: 64×64×32) and Depthwise 3×3 (weight repacking overhead) are severely memory-bound. Their RAM/MACC ratio is 100–1000× higher than standard convolutions, exposing an architectural inefficiency in how the NPU handles these patterns.

⑤ Estimation Accuracy Is Layer-Dependent

Static compiler predictions are accurate for compute-bound standard convolutions (Δ% ≈ 0%) but fail badly for memory-bound layers: Depthwise 3×3 (+100%), Pointwise 1×1 (+900% on some tools). This makes capacity planning unreliable for MobileNet-style architectures.

⑥ Flash Is Compiler-Independent

Flash usage for model weights is identical across all four platforms for the same operator — confirming that INT8 quantization produces the same binary weight layout regardless of compiler version, and Flash footprint is a fixed property of the model.

🛠️Optimization Opportunities

🔄

① Layer Architecture — Replace Pointwise 1×1

Pointwise convolutions generate enormous output activations (64×64×32 = 131K values). Replace with grouped convolutions or use output striding to reduce activation size and RAM pressure. Consider MobileNetV3-style Squeeze-and-Excite blocks instead.

📐

② Input Resolution Reduction

All benchmarks use a 64×64×3 input. Reducing to 32×32 would cut activation tensors by 4× — bringing even 7×7 convolutions within single-epoch range and enabling v4.0 success.

⚙️

③ Compiler Tuning — AI Core v4.0 Epoch Controller

Investigate --epoch-controller CLI flags in ST Edge AI Core v4.0 to explicitly set epoch boundaries for the 7×7 case. The v2.2 compiler handles 4-epoch tiling successfully — this should be replicable in v4.0.

🔗

④ Operator Fusion — Conv + MaxPool

The Conv2D + MaxPool benchmark already demonstrates NPU operator fusion (same latency as standalone Conv2D 16f 3×3: 0.079–0.088 ms). Extend fusion to Conv → BN → ReLU chains to amortise overhead and reduce memory round-trips.

📦

⑤ Depthwise Memory Alignment Fix

The compiler doubles RAM for Depthwise 3×3 due to weight repacking for NPU vector alignment. Use compiler flags to enforce aligned weight layouts at export time (e.g. in ONNX export) to avoid the repacking step and halve its RAM usage.

📊

⑥ Static Estimation Improvement

RAM predictions for memory-bound layers (1×1, depthwise) are inaccurate by up to +1050%. Work with the ST toolchain to improve static analysis by accounting for output activation size and weight alignment padding during the estimation phase.

🏗️

⑦ Architecture-Aware Design — NPU-Friendly Models

Prefer NPU "sweet-spot" operators: standard Conv2D 3×3/5×5 with ≤64 filters. These deliver single-epoch execution, accurate estimates, and near-peak throughput. Depthwise separable convolutions need special attention to avoid the memory-boundedness trap.

🔬

⑧ Extended Benchmarking

Extend tests to: multi-layer networks (fusion effects), FP16/mixed precision, varying batch sizes, and the Cortex-M55 Helium baseline — to fully characterise the NPU vs. CPU trade-off space.

✅Conclusions & Contributions

What Was Achieved

  • Characterised 8 fundamental deep learning operators on the STM32N657 Neural-ART™ NPU across 4 platform configurations.
  • Quantified the compiler generation gap: AI Core v2.2 consistently outperforms v4.0 on the Nucleo board for most operators tested.
  • Identified the 1.5 MB SRAM wall as the primary physical constraint for single-layer deployment, causing FAIL on v4.0 for the 7×7 operator.
  • Exposed RAM estimation inaccuracies of up to +1050% for memory-bound operators (1×1 pointwise) — critical for safe system design.
  • Demonstrated that Flash footprint is compiler-independent and determined solely by model quantization.

Practical Guidelines for STM32N657 Deployment

  • Prefer standard Conv2D 3×3–5×5 with ≤64 filters: single-epoch, predictable, high throughput.
  • Use CubeMX/X-CUBE-AI (AI Core v2.2) as the reference compiler — most reliable and fastest for production.
  • Avoid operators with activation tensors > 1.4 MB to prevent epoch proliferation or runtime failure.
  • Apply operator fusion (Conv+Pool, Conv+BN+ReLU) to reduce memory round-trips.
