⚙️Neural-ART™ Accelerator — STM32N657

Both the Nucleo-N657X0-Q and the STM32N6570-DK mount the same STM32N657 SoC and therefore share an identical NPU silicon — any performance difference observed between the two boards is purely due to compiler version and toolchain configuration.

Streaming Dataflow Architecture

The Neural-ART NPU uses a streaming processing engine: rather than loading full activation maps into SRAM, data flows continuously through the MAC array from input to output, dramatically reducing memory bandwidth requirements.

Key Deployment Constraint

The internal NPU activation buffer is limited to ~1.5 MB. Layers whose activation tensors exceed this limit require multi-pass execution or fail to compile entirely. This is the primary constraint observed across all CV model deployment attempts.

🔢Hardware Epoch Partitioning per Layer & Platform

What Drives Epoch Count?

The compiler partitions a layer into N epochs when the activation buffer required exceeds the internal ~1.5 MB SRAM window. Each additional epoch introduces a reload-and-store cycle, increasing latency.

1️⃣

Most standard convolutions (16f 3×3, 32f 3×3, 64f 3×3, 5×5) fit in a single epoch on all platforms — excellent NPU utilization.

2️⃣

Depthwise 3×3 is split into 2–3 epochs depending on the platform: the compiler over-allocates RAM due to weight-repacking for NPU alignment, forcing partitioning unnecessarily.

2️⃣

Pointwise 1×1 requires 2 epochs on most platforms despite low MACC count — its large output activation map (64×64×32) is the bottleneck.

4️⃣

Conv2D 32f 7×7 requires 4–5 epochs on v2.2 platforms due to 1,292–1,548 KiB activations; v4.0 times out entirely.

📐RAM Estimation Error Δ% (Actual − Estimated) / Estimated

Estimation Accuracy Analysis

Δ% = (Actual − Estimated) / Estimated × 100
Positive = compiler under-estimates (risky — may cause OOM).
Near-zero = accurate prediction.

⬆️

Depthwise 3×3 (CubeMX): actual RAM is 100% higher than estimated — compiler doubles RAM due to weight-repacking alignment overhead for the NPU. This is a known compiler artifact.

🔴

Pointwise 1×1 (AI Studio / DevCloud v4.0): compiler estimates 12 KiB but actual is 130–140 KiB (+900% to +1050%). Streaming output activations are severely underestimated.

✅

Standard convolutions (3×3, 5×5, 64f 3×3): Δ% is ≈ 0% across all tools — the compiler predicts RAM correctly for typical compute-bound layers.

ℹ️

7×7 layer: estimation is accurate (near 0%) — the model is too large by design, not due to compiler error.

🚀Throughput — MACC/ms (NPU Utilisation Proxy)

Why Throughput Matters

Raw inference time is model-size-dependent. Throughput (MACC/ms) normalises by complexity and reveals true NPU utilisation — how many operations per millisecond the hardware is actually executing.

✅

CubeMX v2.2 delivers the highest throughput for compute-bound layers like 5×5 and 7×7, consistent with higher NPU utilisation observed across the v2.2 compiler backend.

📉

Pointwise 1×1 and Depthwise 3×3 show dramatically lower throughput on all platforms — these layers are memory-bound, not compute-bound, and the streaming engine cannot fully pipeline due to large output tensors.

📊

Standard Conv (3×3, 5×5, 64f 3×3) achieve the best MACC/ms — close to the theoretical peak — confirming that regular convolutions are the NPU's optimal operating range.

🔬CV Model Deployment Audit — STM32N657 Hardware

Ten state-of-the-art CV models were exported from PyTorch, INT8-quantized via the ST toolchain, and compiled for the STM32N657 NPU. None of the provided models compiled successfully at native resolution due to three primary constraints:

💥

① Activation Overflow (RAM Too Low)

Models at native resolution (640×640) generate activation tensors exceeding the npuRAM banks (448 KB each). The NPU refuses to compile: Model does not fit in Board. RAM Size too low.

⚙️

② Large-Kernel im2col Explosion (ResNet34)

ResNet34 uses a 7×7 stem convolution. The Neural-ART compiler expands this via im2col, making RAM requirements balloon: Warning: Oauto did not find valid compile options: aborting.

🚫

③ Unsupported Operators — Transformer Layers

MobileCLIP-S0 and MobileCLIP-B Transformer components (LayerNorm, Self-Attention) crash both compilers: INTERNAL ERROR: ONNX shape inference fails or NOT IMPLEMENTED: Unknown layer format.

✅Validated Deployment Strategies

Motivation: Input Resolution Scaling

To isolate failures, CNN models were downscaled to 224×224. This reduces maximum activation tensors from ~1.6 MB to ~200 KB, keeping them within a single 448 KB npuRAM bank. Result: Successful compilation and real-time inference (48.3 FPS).

🔍NPU Behaviour — Key Findings

① Compiler Generation Matters

AI Core v2.2 (CubeMX) outperforms v4.0 (AI Studio) on the Nucleo board for most operators — backend improvements are hardware-target specific.

② 1.5 MB SRAM Hard Limit

The internal buffer (~1.5 MB) is the primary constraint. Overflows require epoch partitioning or cause compiler FAIL (e.g., 7×7 convolution).

③ Standard Conv2D is Optimal

Standard convolutions (3×3, 5×5) achieve near-peak throughput, accurate RAM prediction, and reliable single-epoch execution.

④ Memory-Bound Layers are Costly

Pointwise 1×1 and Depthwise 3×3 suffer from high RAM/MACC ratios, exposing architectural inefficiencies in streaming output activations.

⑤ Estimation Accuracy Is Layer-Dependent

Predictions are excellent for compute-bound kernels (Δ% ≈ 0%) but fail drastically for memory-bound layers (e.g., Pointwise +1050%).

⑥ Fixed Weight Flash

Flash usage is compiler-independent for INT8, determined solely by model quantization density.

🛠️Optimization Opportunities

🔄

① Reduce Pointwise 1×1 Activations

Replace with grouped convolutions or apply output striding to reduce large activation tensors that trigger memory bottlenecks.

📐

② Input Resolution Reduction

Confirmed by CV model tests: reducing input from 640×640 to 224×224 cuts activation tensors by ~8×, enabling successful compilation and real-time inference on the STM32N657 NPU.

⚙️

③ Replace 7×7 Kernels with Two 3×3 in Series

A 7×7 convolution fails on v4.0 even with Epoch Controller enabled. The architectural alternative — two stacked 3×3 convolutions — achieves the same receptive field, fits in a single epoch, and is confirmed compatible on all four platforms.

🔗

④ Extend Operator Fusion Chains

Conv2D + MaxPool fusion is already demonstrated (comparable latency to standalone Conv2D). Extend fusion to Conv → BN → ReLU chains to further reduce memory round-trips and inter-operator overhead.

📦

⑤ Depthwise Weight Alignment at Export

The compiler doubles RAM for Depthwise 3×3 due to weight repacking for NPU vector alignment. Enforcing aligned weight layouts in the ONNX export avoids the runtime repacking step, cutting its RAM usage by ~50%.

🏗️

⑥ Architecture-Aware Model Design

Prefer standard Conv2D 3×3/5×5 with ≤64 filters: single-epoch execution, accurate RAM estimates, and near-peak throughput. Avoid kernels ≥7×7 and Transformer layers — confirmed incompatible with the Neural-ART NPU.

✅Conclusions & Contributions

What Was Achieved

  • Characterised 8 fundamental deep learning operators on the STM32N657 Neural-ART™ NPU across 4 platform configurations.
  • Quantified the compiler generation gap: AI Core v2.2 consistently outperforms v4.0 on the Nucleo board for most operators tested.
  • Identified the 1.5 MB SRAM wall as the primary physical constraint — root cause of all CV model deployment failures.
  • Exposed RAM estimation inaccuracies of up to +1050% for memory-bound operators (pointwise 1×1) — critical for safe capacity planning.
  • Validated that CNN-based models at 224×224 (MobileNetV2, YOLOv8-n, YOLOv11-n) compile and run successfully — confirming input resolution as the key constraint.

Practical Guidelines for STM32N657 Deployment

  • Prefer standard Conv2D 3×3–5×5 with ≤64 filters: single-epoch, predictable, high throughput.
  • Keep max activation tensors < 448 KB per npuRAM bank — determines feasibility more than model architecture.
  • Avoid kernels ≥7×7, Transformer layers, and dynamic ONNX shapes — all confirmed incompatible.
  • Apply operator fusion (Conv+Pool, Conv+BN+ReLU) to reduce memory round-trips.
